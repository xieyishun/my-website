## 项目缘起

我一直挺喜欢 **Loop Hero** 那个核心循环：你不直接控制角色，你只是**给世界放东西**——放石头、放村庄、放墓地，然后看英雄在自己设定的路径上跑、打、死、重开。

它的妙处在于把"玩法"和"操作"完全解耦：玩家做的是**布局决策**，不是手速。

但 Loop Hero 有个我个人觉得"差一口气"的地方：**它的世界观是黑暗奇幻**，所以你放的东西本质都是"让英雄受苦"。每一个布局决策都要在"加资源"和"加难度"之间精心权衡——这非常硬核但也非常累。

我想换一个对应感更强的世界观——**修仙**。

修仙的核心是什么？**布阵、炼丹、坐关、出关**。这个东西天然就是"玩家不直接出手、靠环境系统帮自己变强"的逻辑。聚灵阵让灵气更浓、镇妖塔自动除魔、丹炉自动回血——这些原本就是修仙小说里随处可见的设定，套到 Loop Hero 的循环上几乎是天然契合。

于是有了《轮回阵途》：**布阵改变轮回**。

## 整体架构：双层循环

游戏分两个主场景：

### 1. 局内（Main.tscn）：一次"轮回"

这是 Loop Hero 那一层：玩家修士自动在一个循环道路上跑，每经过一格触发不同效果。玩家**唯一能做的**是在道路周围的格子里放建筑：

| 建筑 | 效果 |
|------|------|
| 聚灵阵 | 范围内 2 格道路变修炼格，跑过得修为 |
| 镇妖塔 | 范围内 1 格变战斗格，跑过触发战斗 |
| 丹炉 | 范围内 2 格变回复格，跑过回血 |
| 悟道碑 | 不改格子类型，但范围内 +50% 收益（可叠加） |
| 灵药田 | 被动：每放一座 +8 最大 HP |
| 灵木 | 被动：每放一座 +5% 修为收益 |
| 矿石 | 被动：每放一座 +6% 灵石掉落 |

这里就出现了第一个有意思的策略点：**前三种是"会改地形"的，后四种是"纯加成"的**。前者会推动战斗节奏（更多战斗 = 更多经验 + 更多死亡风险），后者纯堆数值。这种"主动型 vs 被动型"的二元划分，本质上是逼玩家选择"激进派"还是"积累派"。

### 2. 局外（Sect.tscn）：宗门五行布阵

每次"轮回"结束（死亡或主动重入），玩家带着碎片和材料回到宗门。宗门是一个 **5×5 的格子**，玩家在上面放五种五行建筑：

- 🌳 **青木殿（木）** → +12 初始 HP
- 🔥 **烈焰塔（火）** → +2 初始攻击
- ⛰️ **后土坛（土）** → +1 初始防御
- ⚱️ **真金阁（金）** → +6% 灵石掉落
- 💧 **玄水池（水）** → +6% 修为收益

关键在于**相邻关系**：

> **相生（+25%）**：木→火→土→金→水→木
> **相克（-20%）**：木克土，土克水，水克火，火克金，金克木

放置的位置会决定每个建筑的**实际增益**——把烈焰塔放在青木殿旁边（木生火），它就强 25%；放在玄水池旁边（水克火），它就弱 20%。

这是整个游戏的**长期养成核心**。它把 Loop Hero 没有的"宏观规划层"补上了：**每一局的强度，是上一局结束后你在宗门里布阵决定的**。

<InsightCallout>
两层循环的设计目标：让"局内策略"和"局外规划"形成正反馈——局内多攒资源 → 局外能解锁更好的五行布阵 → 下一局起步就更强 → 走得更远 → 攒更多资源。
</InsightCallout>

## 已完成的系统

到目前为止，代码层面完成度我自己估个 60%。把 `Main.gd` 打开数了一下，**14 个 Manager** 已经全部跑通：

```text
GridManager → RoadManager → MapRenderer
↓
CardManager / EquipmentManager / InventoryManager（独立）
↓
BuildingManager（依赖 Grid + Card）
↓
PlayerManager（依赖 Grid + Road + Equipment）
↓
BattleManager（依赖前面所有）
↓
EventManager / MonsterSpawner / TimeBarManager / EventBarManager / BossManager
```

加上 6 个全局 autoload：

- **SignalBus** —— 所有跨模块通讯走全局信号，避免硬耦合
- **Cultivation** —— 境界查询（练气 / 筑基 / 金丹 / 元婴 / 炼神 / 返虚 × 三阶 + 大圆满）
- **DataLoader** —— JSON 数据加载
- **SaveManager** —— 存档（材料 / 灵石 / 碎片 / 宗门布局）
- **RunInventory** —— 当局背包
- **GameManager** —— 游戏状态

### 数据全部 JSON 化

我一开始就强制自己**所有可调数值都写在 JSON 里**——目前有 6 份数据：

- `buildings.json` —— 7 种局内建筑
- `sect_grid_buildings.json` —— 5 种五行宗门建筑
- `equipment.json` —— 6 个境界 × 多个槽位的装备
- `enemies.json` —— 4 种怪物
- `events.json` —— 4 个随机事件
- `sect_buildings.json` —— 宗门基础建筑

这个决策我现在回头看是**完全正确**的。每次平衡数值不用动代码、不用重启编辑器，改 JSON 重新跑一局立刻验证。

## 踩过的坑

### 坑 1：相机适配 UI 占用区域

Loop Hero 的地图本身不大，但**周围的 UI 占了非常多屏幕**——顶部状态栏、右侧装备/背包、底部手牌/日志。如果相机直接盯死地图中心，地图会被 UI 遮一半。

最后的解法在 `Main.gd::_position_camera()` 里——根据视口大小**反算可视区域中心**，再把相机偏移到那个点：

```gdscript
var visible_w = window_size.x - ui_left_margin - ui_right_margin
var visible_h = window_size.y - ui_top_margin - ui_bottom_margin
var zoom = clampf(minf(visible_w / grid_w_px, visible_h / grid_h_px), 0.35, 1.0)
```

不优雅但能用。后续可能要换成 `SubViewportContainer`，让地图在自己的 viewport 里独立缩放。

### 坑 2：暂停时还要能放建筑

Loop Hero 的核心爽点之一是**"我在思考的时候世界要等我"**——但 Godot 的 `process_mode` 默认下暂停整棵节点树，鼠标输入也会被吃掉。

最后用的方案是分层：

- `Main.gd` 设 `PROCESS_MODE_ALWAYS`（暂停时仍能接收点击事件）
- `PlayerManager` / `BattleManager` / `MonsterSpawner` 设 `PROCESS_MODE_PAUSABLE`（暂停时该停的还停）

这样暂停 = 只暂停"会动的东西"，玩家依然能放/拆建筑。这个解耦花了我一个晚上。

### 坑 3：境界系统别一上来就大而全

最早我想做的境界系统是**每个境界都有独立的特性、独立的天赋树**——结果发现还没等做完一个，整个系统已经膨胀到难以平衡了。

后来狠心简化成 `Cultivation.gd` 现在这样：**纯查询函数**，给一个 level 返回境界名。境界本身不带额外效果，只是一个**叙事标签 + 装备解锁门槛**（装备的 `realm` 字段对应境界等级）。

**真正影响数值的是 level 本身**，境界只是"换个好听的说法"。这样代码极简，但叙事感完全保留。

## 一些感想

做了一个月发现，Loop Hero 这个范式真的特别适合**单人小项目**：

- 玩家不直接控角色 → **不用做角色操作动画**，节省了一大半美术量
- 自动战斗 → **不用做技能 UI 和释放反馈**
- 主要交互是"放/拆方块" → **UI 输入路径极简**

剩下能投入精力的地方就回到了**系统设计本身**——格子之间的相互作用、建筑的相互组合、长期养成的反馈节奏。这正好是我感兴趣的部分。

而修仙这层皮，比我预期更契合——**境界、相生相克、布阵、坐关**，每一个机制点都能在世界观里找到对应的"修仙说法"。这种"系统命名自带正当性"的舒服感，真的会让开发节奏快很多。
